# 附身模组实践计划

## 目标

做一个 Minecraft 1.20.1 Fabric 模组，核心体验是：

- 玩家可以对准生物发起附身
- 附身后获得目标实体视角
- 能主动退出附身
- 行为上尽量贴近原版实体与旁观者视角的运行方式
- 技术上兼容原版玩家、实体、传送、死亡、多人同步逻辑

## 总原则

1. 不替换玩家底层生命周期，优先保留原版玩家实体。
2. 先做“视角附身”，再做“控制附身”，最后做“能力附身”。
3. 服务器权威，客户端只负责输入、视角和表现。
4. 附身状态必须能在死亡、传送、换维度、目标死亡时自动回收。
5. 每一步都要和原版逻辑对齐，避免和原版代码打架。
6. 附身期间会对被附身生物施加增益，玩家生命值以当前被附身生物的增益后生命值为准。

## 阶段划分

### 阶段 0：工程骨架

目标：

- 建立 Fabric 1.20.1 工程
- 跑通客户端和服务端初始化
- 确认能编译、能启动、能进世界

产出：

- `modid`
- 基础初始化类
- 客户端初始化类
- 网络包注册位置
- 配置和日志骨架

验收：

- 游戏能正常启动
- 模组能被识别
- 日志输出清晰

---

### 阶段 1：视角附身 MVP

目标：

- 玩家按键对准实体后请求附身
- 服务器校验后批准
- 客户端将镜头切到目标实体
- 玩家可以手动退出，镜头恢复自身

实现点：

- 客户端按键绑定
- 准星实体检测
- C2S 附身请求
- S2C 批准或拒绝
- `setCameraEntity` 切换相机
- 退出附身请求

验收：

- 对准一个动物按键后，镜头切过去
- 按退出键后，镜头切回玩家
- 距离不够或目标无效时不会附身

---

### 阶段 2：附身状态管理

目标：

- 服务器维护一个附身会话
- 处理自动退出和异常回收
- 避免状态卡死

状态字段建议：

- `playerUuid`
- `targetUuid`
- `startTick`
- `lastValidateTick`
- `reason`
- `cooldownUntil`

回收条件：

- 玩家死亡
- 目标死亡
- 目标卸载
- 玩家传送
- 换维度
- 距离超过上限
- 玩家主动退出

验收：

- 任意异常都不会把相机卡死
- 退出后状态可重新进入
- 冷却生效

---

### 阶段 3：HUD 和表现层

目标：

- 让玩家明确知道自己正在附身
- 提供退出提示和状态信息

HUD 内容建议：

- 当前附身对象名称
- 附身持续时间
- 稳定度或冷却状态
- 退出按键提示

表现层建议：

- 附身进入时播放音效或轻特效
- 退出时播放回退特效
- 视角高度按目标实体调整

验收：

- 玩家能一眼看出自己处于附身状态
- 不看聊天栏也知道怎么退出

---

### 阶段 4：原版适配层

目标：

- 让附身逻辑服从原版实体规则
- 不破坏原版交互、传送、伤害、死亡逻辑

重点适配：

- 玩家死亡与重生
- 目标实体死亡
- 下界门、末地门、命令传送
- 骑乘状态
- 潜行、游泳、攀爬
- 药水效果和伤害来源

原则：

- 原版已有规则能复用就复用
- 只有必要时才用 Mixin 补逻辑

验收：

- 普通世界玩法不被破坏
- 附身时不会造成大量原版 bug

---

### 阶段 5：能力映射

目标：

- 不同生物拥有不同的附身表现
- 让附身不仅是视角切换，而是“身体变化”
- 附身时同步施加生物增益，并让玩家生命值跟随增益后的目标生命值

建议先做的第一批：

- 蜘蛛：攀爬
- 海豚：水下移动增强
- 蝙蝠：低体型与侦察感
- 狼：追踪与轻战斗
- 骆驼：沙漠移动

先不做：

- 监守者
- 末影龙
- 凋灵
- 村民交易类深逻辑

增益规则建议：

- 进入附身时，给目标形态套用一组预设增益
- 增益可以作用于最大生命值、移动速度、攻击伤害、击退抗性、游泳速度等
- 玩家当前生命值以“增益后的最大生命值”为上限和基准
- 若退出附身，生命值按规则折回原状态，避免溢出或瞬死

验收：

- 不同形态确实有明显差异
- 不会因为一个形态把整个系统拖崩

---

### 阶段 6：扩展与平衡

目标：

- 增加更多可附身生物
- 做稳定度、代价、冷却、稀有度
- 形成完整玩法循环

平衡手段：

- 强形态高代价
- 强形态长冷却
- 稀有形态需要前置条件
- 不适配形态有弱化或限制

验收：

- 玩家会思考什么时候附身、附身谁
- 不会形成“最强形态一把梭”的局面

## 技术拆分

### 客户端

- `KeyBindingHelper`
- 附身按键
- 退出按键
- HUD 渲染
- 相机切换
- 本地特效

### 服务端

- 附身请求校验
- 附身会话管理
- 自动退出判定
- 冷却判定
- 生物适配规则
- 目标生物增益应用
- 增益后生命值同步

### 网络

- `C2S_AttachRequest`
- `C2S_DetachRequest`
- `S2C_AttachAccepted`
- `S2C_AttachRejected`
- `S2C_AttachForceExit`

### 原生兼容点

- 目标实体状态检查
- 玩家生命周期事件
- 服务器 tick 检查
- 维度切换处理
- 伤害与死亡回收
- 目标生命值属性同步
- 附身结束后的生命值折算

## 第一版最小目标

第一版只完成：

- 选中实体
- 请求附身
- 服务器校验
- 切换镜头
- 退出附身
- HUD 提示
- 自动回收

这是最小可玩版本。

## 风险点

1. 直接替换原版 mob 实体，风险很高。
2. 不做服务器权威，联机会乱。
3. 试图一次支持所有生物，工程量会爆炸。
4. 忽略死亡、传送、换维度，容易卡死状态。

## 下一步执行顺序

1. 确定 modid、包名、基础工程结构。
2. 搭建客户端按键和网络包。
3. 先实现镜头附身和退出。
4. 加状态管理和自动回收。
5. 再做 HUD 和原版适配。
6. 最后扩展生物能力。
